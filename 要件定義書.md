# 宿泊施設ベース価格改定自動化ツール：詳細業務定義・要件定義書（完全版）

## 1. プロジェクト概要
東京都内約130施設の宿泊施設を運営する株式会社エアステイにおいて、月次で実施する「ベース価格の見直し」をデータドリブンかつ自動的に行うためのシステム仕様。

### デプロイURL
- **本番環境（Cloudflare Pages）**: https://base-price-coordination-tool-9yr.pages.dev/

---

## 2. 入力データの詳細定義

### ① ベース価格推移マスタ（`価格調整の推移記録.xlsx` または CSV）

各物件の価格履歴を管理する。**Excel形式（.xlsx）とCSV形式の両方に対応**。

#### Excel形式（推奨）
実運用で使用する `価格調整の推移記録.xlsx` をそのままアップロードできる。

- **対象シート**: `各物件単価`
- **構造**:

| 列 | 内容 |
|---|---|
| A列 | 行番号（平日行のみ入力、週末行は空欄） |
| B列 | 物件名（平日行のみ入力、週末行は空欄） |
| C列 | 日区分（「平日」または「週末」） |
| D列（当初） | 運用開始時の初期価格 |
| E列以降 | 価格変更日（yyyy/m/d形式）とその時点の価格 |

- **疎データ形式**: 価格変更があった日付の列にのみ値が入る（変更なしの列は空欄）。「現在価格」は最右端の非空欄セルの値として取得する。
- **撤退物件**: いずれかの日付セルに「撤退」と入力されている物件は計算対象から除外する。
- **他シートの保持**: `倍率修正`・`過去_金額修正` など他のシートは出力時にそのまま引き継ぐ。

#### CSV形式（後方互換）
- **必須カラム**: 物件名、日区分、日付列（`YYYY-MM-DD` 形式）
- 全ての日付列に値が入る密形式。

#### 共通仕様
- **管理単位**: 1物件につき「平日」と「週末」の2行。
- **特性**: 毎月の実行後、一番右側に「新しい日付の列」が追加される。

---

### ② 新規予約実績データ

過去1〜2ヶ月間に「新しく入った」予約の、宿泊月別件数。**PMS出力形式とM1-M10形式の両方に対応**。

#### PMS出力形式（推奨）
AirHostなどのPMSから「**過去1〜2ヶ月に確定した予約**」の条件でエクスポートしたCSVをそのままアップロードできる。2ヶ月分まとめてアップロードしても正しく集計される。  
ファイル名例：`Booking_20260502172043.csv`

- **PMSエクスポート条件**: 予約確定日（予約日）を過去1〜2ヶ月で絞り込んでエクスポートする。ツール側での予約日フィルタは不要（チェックイン月のみで判定）。
- **必須カラム**: `物件名`、`チェックイン`、`状態`
- **自動集計ロジック**（実行月設定をもとに自動処理）:

  | ステップ | 処理内容 |
  |---|---|
  | ① ブロック除外 | `状態` = `ブロックされた` の行を除外する |
  | ② キャンセル除外 | `状態` に「キャンセル」を含む行を除外する |
  | ③ リスティングキー生成 | 後述「多部屋物件の管理ルール」に従い、`物件名`・`部屋番号` 列からリスティングIDを生成する |
  | ④ M1-M10マッピング | `チェックイン` 月を基準に実行月翌月（M1）〜10ヶ月先（M10）に振り分ける。それ以外のチェックイン月は集計対象外 |
  | ⑤ リスティング別集計 | リスティングIDごとにM1〜M10の件数を合計し、内部的にM1-M10形式に変換する |

- **除外される行の例**:
  - 実行月（5月）および前月（4月）チェックイン → 過去・実行月のため対象外
  - 11ヶ月先以降のチェックイン → M10を超えるため対象外
- **処理結果プレビュー**: アップロード後、集計内訳（読込総数・ブロック除外・キャンセル除外・範囲外除外・集計対象件数と物件数）と物件別M1-M10テーブルをUI上に表示する。

#### M1-M10形式（後方互換）
- **必須カラム**: 物件名、M1〜M10（実行月の翌月をM1、その次をM2とした10ヶ月先までの予約件数）
- 例：5月実行の場合、M1=6月、M2=7月...の予約件数。

---

### ③ 現在の予約状況データ【任意】

ツール実行時点で将来の日程にすでに入っている予約の宿泊月別件数。

- **目的**: 既に予約が多く入っている物件に対して、新規予約の動きが鈍かった場合でも誤って値下げしてしまうことを防止する。
- **任意項目**: このデータはアップロード省略可。省略時は新規予約実績データのみでスコアを算出する。

#### AirHost出力形式（推奨）

AirHostから「**将来チェックインの全予約**」を期間別にエクスポートしたCSVをそのまま使用できる。**ファイルは複数に分かれていてもよい**。

- **想定エクスポート構成**: 対象12か月（例：5月〜翌年4月）を2か月ごとに分割した6ファイル
  - `Booking_YYYYMMDDHHMMSS.csv`（AirHostの標準ファイル名形式）
- **必須カラム**: `物件名`、`チェックイン`、`状態`（② 新規予約実績データと同じ列構造）
- **アップロード方法（2通り）**:

  | 方法 | 操作 |
  |---|---|
  | **ファイル選択** | 最大6ファイルをまとめてドラッグ＆ドロップ（または複数選択） |
  | **テキスト貼り付け** | 各ファイルの内容をテキストエリアに順番に貼り付け → 「処理する」ボタン押下 |

- **複数ファイルの結合処理**: ファイルをまたいで発生するヘッダー行の重複は自動で除去し、全行を結合してから集計する。BOM付きCSVにも対応。
- **自動集計ロジック**: ② 新規予約実績データと同一（ブロック除外→キャンセル除外→M1-M10マッピング→物件別集計）
- **処理結果プレビュー**: アップロード後、集計内訳と物件別M1-M10テーブルをUI上に表示する（緑背景）。

#### M1-M10形式（後方互換）
- **必須カラム**: 物件名、M1〜M10（新規予約実績データと同じ月区分で、現時点での累積予約件数）

## 3. 計算ロジックの詳細定義

### ① 加重スコア（LT考慮スコア）の算出
将来の予約が「いつ」入っているかにより、需要の強さを数値化する。直近の予約ほど価値を高く設定する。

**加重係数（デフォルト設定）**:
- M1（翌月）: 件数 × 8点
- M2（2ヶ月先）: 件数 × 6点
- M3（3ヶ月先）: 件数 × 4点
- M4（4ヶ月先）: 件数 × 2点
- M5〜M10（5〜10ヶ月先）: 件数 × 1点

**計算式**:
```
Score = (M1×8) + (M2×6) + (M3×4) + (M4×2) + (M5+M6+M7+M8+M9+M10)×1
```

**現在の予約状況データがある場合のスコア補正**:
- 上記の計算式を新規予約実績データ（②）と現在の予約状況データ（③）の両方に対して個別に適用し、それぞれ「新規スコア」「現況スコア」を算出する。
- 最終的な加重スコアは、**両スコアの高い方**を採用する。
  ```
  最終スコア = max(新規スコア, 現況スコア)
  ```
- **採用理由**: 先月の新規予約は少なくても、過去に積み上がった予約がすでに多い物件は需要が高いと判断し、値下げを防止するため。
- 現況スコアが採用された場合、提案レポートの備考欄にその旨を記録する。

### ② 価格改定判定ロジック
算出されたスコアを以下のマトリクスに当てはめ、**現在のベース価格（最新の列の値）**に対して増減を行う。

| 加重スコア（S） | 変動額（平日・週末共通） | ステータス判定 |
|:---|:---|:---|
| S ≥ 81 | **+1,000円** | 超高需要：即値上げ |
| 41 ≤ S ≤ 80 | **+500円** | 高需要：強気設定 |
| 21 ≤ S ≤ 40 | **±0円** | 適正需要：価格維持 |
| 1 ≤ S ≤ 20 | **-500円** | 低需要：弱気設定 |
| S = 0 | **-1,000円** | 需要なし：大幅調整 |

## 4. 新規物件の追加プロセス（自動拡張要件）

ツール実行時、以下のフローで新規物件をマスタに組み込む。

1. **照合**: 予約実績データにある物件名が、価格推移マスタに存在するか確認。
2. **新規検知**: マスタに存在しない物件名がある場合、自動的にマスタの末尾に「平日」「週末」の2行を追加する。
3. **初期価格の設定（UIパネル）**:
   - 新規物件が検出されると「新規物件の初期価格設定」パネルが表示され、各物件の平日・週末価格を手動入力できる。
   - 入力後、以下の2つのボタンのいずれかで確定する：

   | ボタン | 動作 |
   |---|---|
   | **確定してレポートに反映する（初期価格のまま）** | 入力した価格をそのまま新価格として使用する。需要スコアによる価格調整は行わない。 |
   | **確定して需要スコアで価格提案する** | 入力した価格を「旧価格（ベース）」として、予約実績データから需要スコアを計算し、スコアに応じた価格増減を適用した提案価格を算出する。 |

4. **Google Sheetsへの書き出し**:
   - 新規物件はマスタシートの適切な位置に「平日」「週末」の2行として挿入される。
   - A列（連番）も自動で入力され、後続の既存物件の連番も繰り下がる。
   - 最新日付列に初期価格（または需要スコア適用後の提案価格）が書き込まれる。
5. **継続管理**: 翌月以降は他の物件と同様に、前月の予約実績に基づいた価格改定ループに入る。

## 5. 出力データの詳細定義

### ① 更新済みベース価格推移マスタ

入力ファイルのフォーマットに応じて出力形式が変わる。

| 入力形式 | 出力ファイル名 | 出力形式 |
|---|---|---|
| Excel（.xlsx） | `New_Master_Price_{yyyymm}.xlsx` | 元のExcelと同じ構造。`各物件単価` シートの右端に新日付列を追加。`倍率修正` 等の他シートも保持。 |
| CSV | `New_Master_Price_{yyyymm}.csv` | 元のCSVの右端に新日付列を追加。 |

- 元のマスタの右端に、実行月の1日（例：`2026-05-01`）の列を追加する。
- 計算された「新ベース価格」を全ての物件（既存＋新規）に対して書き込む。
- 撤退物件は新列への書き込みを行わない（空欄のまま）。

#### 価格変化による文字色分け（Excel出力・Google Sheets出力 共通）
新しく追加した列のセルを、前回価格との比較結果に応じて以下の色で色分けする。

| 状態 | 文字色 |
|---|---|
| 前回より**高くなった**（値上げ） | 青（`#1155CC`） |
| 前回より**低くなった**（値下げ） | 赤（`#CC0000`） |
| **変わらなかった**（価格維持） | 黒（`#000000`） |

- 「前回価格」は同じ行の右端から左方向に探索し、最初に見つかった有効な価格を使用する。
- 前回価格が存在しない行（新規物件等）は黒で表示する。

### ② ベース価格提案レポート（`Proposal_Report_{yyyymm}.csv`）
現場担当者が確認するためのサマリー。
- **カラム（基本）**: 物件名, 日区分, 旧価格, 加重スコア（最終）, 変動額, 新価格, ステータス, 備考（新規物件フラグ等）
- **カラム（現在の予約状況データあり時に追加）**: 新規予約スコア, 現況スコア, 現況スコア採用（◯/空欄）

## 6. 例外処理・ガードレール
- **価格の下限設定**: どんなに予約が入らなくても、あらかじめ設定した「最低販売単価（例：6,000円）」を下回らないように制御する。
- **データ欠損**: 予約実績データに物件名がない（＝先月の新規予約が0だった）場合は、スコア0（-1,000円）として処理する。
- **撤退物件**: マスタ上で物件名または任意のセル値に「撤退」が含まれる物件は、計算対象から除外する。

## 7. 月次運用サイクル（イメージ）
1. **毎月1日〜5日（PMSエクスポート）**: AirHostなどのPMSで、**予約確定日（予約日）＝過去1〜2ヶ月**の条件で予約一覧をCSVエクスポートする。2ヶ月に1回の実行など間隔が空いた場合は2ヶ月分をそのままエクスポートして使用可。
2. **ツール実行**:
   - Step 1: `価格調整の推移記録.xlsx`（またはCSV）をアップロード。
   - Step 2: 手順1でエクスポートした `Booking_*.csv` をアップロード。**2ヶ月分のデータもそのまま使用可**。ツールが自動でブロック・キャンセルを除外し、M1-M10のチェックイン月別件数を集計・スコア算出する。
   - Step 3（任意）: AirHostからダウンロードした将来予約CSV（最大6ファイル）をアップロード、またはCSV内容をテキストエリアに貼り付けて「処理する」。ツールが自動でブロック・キャンセルを除外し、M1-M10形式に変換する。
3. **確認**: 提案レポートを見て、異常値がないか専門家が最終チェック。現況スコア採用欄で、どの物件が現況データに基づく補正を受けたか確認できる。
4. **反映**: 更新されたマスタ（Excel）を保存し、宿泊予約システム（PMS）のベース価格を更新する。

- **メインファイル**: `index.html`
- **依存ライブラリ**（CDN経由）:
  - PapaParse 5.4.1（CSV解析）
  - SheetJS 0.18.5（Excel読み書き）
- **入力ファイル**:
  - マスタ: `価格調整の推移記録.xlsx`（推奨）または `Master_Price_History.csv`
  - 予約実績: `Booking_*.csv`（PMS出力形式・推奨）または `Booking_Performance.csv`（M1-M10形式）
  - 現在の予約状況: `Booking_*.csv` × 最大6ファイル（AirHost出力形式・推奨）または M1-M10形式CSV（後方互換）【任意】
- **出力ファイル**:
  - `New_Master_Price_{yyyymm}.xlsx`（マスタがExcel形式の場合）
  - `New_Master_Price_{yyyymm}.csv`（マスタがCSV形式の場合）
  - `Proposal_Report_{yyyymm}.csv`
- **設定項目**:
  - 最低販売単価（デフォルト：6,000円）
  - 実行月（yyyymm形式）
  - 新規物件のデフォルト価格（デフォルト：10,000円）
  - 部屋識別列名（デフォルト：`部屋番号`）：予約CSVの部屋番号が入っている列の見出し名。自動検出に失敗する場合にここで明示指定する。
- **Google Sheets連携**:
  - マスタ読み込み・書き出し先のSpreadsheet URLとして、以下のURLをデフォルト表示する。

## 9. 多部屋物件の管理ルール

一つの物件に複数の部屋（リスティング）がある場合（例：ASAK - AIRSTAY Asakusa #01〜#13）は、**部屋ごとに独立したリスティングとして管理**する。部屋ごとに広さ・価格・需要・SEO・レビューが異なるため、スコア算出・価格改定はすべて**部屋単位（リスティング単位）**で実施する。

### マスタの登録ルール

「物件名」列に**部屋番号を含めた一意の名前**を使用する（`*` は含めない）。

| 例 | マスタの物件名 |
|---|---|
| 一棟マンション型・複数部屋 | `ASAK - AIRSTAY Asakusa #01`、`ASAK - AIRSTAY Asakusa #02`… |
| タイプ別 | `AirStay池袋 Aタイプ`、`AirStay池袋 Bタイプ` |
| 単室物件 | `AMTB - Azuma`（部屋番号なし） |

### AirHostエクスポートCSVのリスティングキー生成ロジック

AirHostのCSVは `物件名`（物件レベル）と `部屋番号`（部屋レベル）が別列で出力される。ツールは以下の優先順位でリスティングIDを自動生成する。

| 条件 | 生成されるリスティングID | 例 |
|---|---|---|
| `部屋番号` が空 | `物件名`（`*` 除去） | `AMTB - Azuma` |
| `部屋番号` が `物件名` で始まる（フルネーム形式） | `部屋番号`（`*` 除去） | `USSJ - Up Support Sangenjaya #304` |
| `部屋番号` が短い部屋コード形式（例：`#01`） | `物件名 部屋番号`（`*` 除去・結合） | `ASAK - AIRSTAY Asakusa #01` |

- **`*` の自動除去**: AirHostの物件名に含まれる `*`（例：`ASAK*`）はマスタには存在しないため、照合時に自動で除去する。
- **部屋識別列名の設定**: デフォルトは `部屋番号`。設定欄で変更可能。自動検出に失敗した場合は赤バッジとともに実際の列名一覧を表示する。
- **照合ミスマッチ診断**: 実行後、マスタに存在するが予約データと一致しなかった物件名と、予約データに存在する物件名を並べて表示し、不一致の原因を特定しやすくする。

### マスタ名とPMS名の整合性

マスタの「物件名」とツールが生成するリスティングIDが**一字一句一致**している必要がある。不一致の場合は「先月の予約実績なし（スコア0）」として処理される。`*` はツールが自動除去するため、マスタには含めない。
